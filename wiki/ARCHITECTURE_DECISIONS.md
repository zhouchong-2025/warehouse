## 决策一:要不要建"物料主数据层",而非按选型册割裂?

### 结论:**应该建。** 这是项目扩展性的地基。

### 实测证据

| 指标 | 数据 |
|------|------|
| 模拟册 PN 数 | 861 |
| 汽车册 PN 数 | 252 |
| **完全相同 PN** | **0** |
| **基础型号族重叠** | **97 个**(LM2903、TPA1882、TPT11695、TPP363070...) |

同一芯片族在两册里是**两条割裂记录**,且命名不同:
- 模拟册: `TPP363070`(工业级,Production)
- 汽车册: `TPP363070Q-FC6R-S`(车规后缀,量产)

更糟:汽车册的 `_section` 还标错了——`TPP363070Q`(实为降压变换器)被标成"数字隔离器",
`TPA8023`(隔离放大器)也标成"数字隔离器"。**按手册组织 = 同一颗芯片的信息分散且互相矛盾。**

### 问题本质

当前数据模型是 **"选型册 → 产品"** 两层:
```
products_structured.json
├── 3peak-analog  (861)  ← 按"册"切
├── 3peak-auto    (252)  ← 同族芯片在这里又来一遍
├── novosense     (1066)
└── yutai         (66)
```
这导致:
- 同族芯片重复、命名漂移、参数口径不一(工业版 vs 车规版分不清是同族)
- FAE 搜"TPP363070"搜不到车规版,搜车规版搜不到工业版
- 没法回答"这颗料有没有车规版本""这个族系有哪些封装选择"

### 建议:三层物料主数据模型(MDM)

```
物料族 (Family)            ← TPP363070 系列
  └── 物料 (Material/SKU)   ← TPP363070、TPP363070Q-FC6R-S(具体订货型号)
        └── 来源 (Source)   ← 出现在哪本册/哪页/原始参数行(可追溯)
```

关键设计:
1. **Family 归一**:用基础型号(去封装/等级后缀)聚合。`TPP363070` 和 `TPP363070Q-FC6R-S` 归同一 family。
   归一规则需 FAE 确认(后缀语义:Q=车规、-S=丝印、-FC6R=封装代码)。
2. **Material 保留每个订货型号**:封装、等级(工业/车规)、状态各自独立,**不合并不丢失**。
3. **Source 可追溯**:每条参数标注来源(册名+页码+原始行),符合你"参数=PDF原文"铁律,出错能溯源。
4. **wiki/entities/ 已是雏形**:现在 `wiki/entities/yutai/YT8011A.md` 这种单料页面,
   就是 Material 层的前身。建议把它升级成正式的 MDM,而不是和 JSON 数据双轨。

### 收益
- 同族芯片统一视图:一次搜索看到工业版+车规版+所有封装
- 跨册参数对账:发现汽车册 section 标错这类矛盾
- 扩展性:第 5、6 家厂商进来,是往 Material 加记录,不是再开一个割裂的册

## 决策二:MinerU 是不是更好的提取工具?

### 结论:**不推荐。放弃 MinerU,用 pymupdf 坐标法。**

### 实测证据(我用你的 key 全程跑通了)

**先澄清:你之前卡的接口问题,根因是 API 版本变了。**
- 旧脚本用 `GET /api/v4/extract/upload-url` → 现在 **404 已废弃**
- 新版正确流程: `POST /api/v4/file-urls/batch` → PUT 上传 OSS → 轮询 `extract-results/batch/{id}`
- 我已验证整条链路可用。**所以不是你不会用,是文档过时了。**

**但跑通后发现 MinerU 不适合本项目:**

裕太 PDF 实测,MinerU(无论 is_ocr=true/false 结果完全一致)输出:
```
正确原文              MinerU OCR结果
130nm          →     130mm        ❌
28nm           →     28mm         ❌
100Base-T1     →     100bas5AT    ❌
RTL9000B       →     RTL9008      ❌
802.3bp        →     80236p       ❌
兼容           →     兼客 / 兼密   ❌
```

**对比 pymupdf 直接抽取同一份 PDF:**
```
YT8011A...QFN48...28nm...MP...802.3bw/802.3bp...P2P兼容RTL9000B系列
```
**字符 100% 精确,一个不错。**

### 为什么 MinerU 在这里帮倒忙

1. **你的 PDF 都有文本层**(可选中文本)。pymupdf 抽的是 PDF 内嵌的精确字符;
   MinerU 走视觉 OCR,把精确文本降级成"看图猜字",必然引入字符错误。
2. **半导体选型对参数字符零容忍**:`28nm→28mm`、`RTL9000B→RTL9008` 是不可接受的。
   这直接违反你的**零假阳性铁律**。
3. **MinerU 唯一的亮点(表格结构识别 `<table>`)解决的是"排版重组"**,
   而这件事用 pymupdf 的坐标信息也能做,且**不损失字符精度**。

### MinerU 什么时候才该用
- 真正的扫描件/图片型 PDF(无文本层)。**目前四份都不是。**
- 如果未来某厂商只给图片版 datasheet,再单独对那一份用 MinerU。

### 正确的提取方向:pymupdf 坐标法

核心:`page.get_text("dict")` 返回每个文字 span 的 **bbox 坐标**。用坐标做二维重建:
- **同一列**:x 坐标相近的 cell 归为一列 → 解决列转置(纳芯微)
- **同一行**:y 坐标相近的 cell 归为一行 → 解决行错位
- **表头合并**:表头区域内 y 相近、x 相近的碎片 span 合并 → 解决表头碎片(模拟册)
- **record 切分**:按 PN 出现的 y 坐标分段 → 解决竖排记录(裕太)

**一套坐标逻辑同时解决表头碎片 / 列转置 / 记录切分三大难题,且字符全部来自原文。**
这才是根因级方案,而不是给 16 个 extract 脚本继续打补丁。

## 决策三:后续交付架构有什么建议?

> 当前交付链路: JSON 全量 load → 前端 JS 文本匹配 → 卡片/对比/CSV。
> 2245 款能撑,但有三个结构性约束限制了 FAE 体验上限。

### 约束 1:参数是字符串,无法做数值范围查询

现在 `_params` 是 `"GBW (MHz): 1"` 这种字符串。FAE 想问"GBW>50MHz 且 Vos<1mV"
做不到,只能靠预生成的离散标签(高速≥50MHz)近似。

**建议:提取阶段就把参数数值化 + 带量纲。**
```json
"params": {
  "gbw": {"value": 50, "unit": "MHz", "raw": "50", "source": "模拟册p9"},
  "vos_max": {"value": 1, "unit": "mV", "raw": "1", "source": "..."}
}
```
保留 `raw` 原文(满足铁律),同时有 `value+unit` 支持范围查询。

### 约束 2:全量内存扫描 + 字符串 includes,无索引

2245 款可接受,上万款(多厂商扩展后)会卡。且字符串 includes 匹配不精确。

**建议分阶段:**
- **近期(<5千款)**:维持前端方案,但把匹配从"字符串拼接 includes"升级成"结构化字段匹配"。
- **中期(上万款)**:引入轻量结构化查询层。推荐 **SQLite(可带 FTS5 全文 + 数值索引)**
  或 **DuckDB**(列存,适合参数范围查询)。前端调 API,不再全量 load。
- 这一步等多厂商铺开后再做,现在不是瓶颈。

### 约束 3:查询理解层与数据层的标签是两套(漂移)

已在 HANDOFF 文档详述:`query_parser.ts`(查询侧)和 `tag_config.py`(数据侧)
各写了一遍同样的阈值。**交付质量取决于两套不漂移,但现在没有机制保证。**

**建议:统一到 tag_schema.json 单一真源,TS 和 Python 都读它**(详见 HANDOFF §4)。

### 推荐的目标交付架构

```
┌────────────────────────────────────────────────────────┐
│ 数据层 (Material MDM + 数值化参数)                        │
│   pymupdf坐标法 → 二维网格 → 列级量纲校验 → 物料主数据      │
│   每参数: {value, unit, raw, source}  可追溯               │
└───────────────────────┬────────────────────────────────┘
                        │ 单一真源 tag_schema.json
                        │ (TS/Python 共读, 规则声明式)
                        ▼
┌────────────────────────────────────────────────────────┐
│ 查询理解层 (规则解释器, 非23个补丁)                        │
│   query → schema规则(mutex/enum/requires) → 结构化条件     │
└───────────────────────┬────────────────────────────────┘
                        │
                        ▼
┌────────────────────────────────────────────────────────┐
│ 匹配/交付层 (结构化查询, 近期前端/中期SQLite)              │
│   数值范围 + 标签精确匹配 → 产品族视图 + 参数对比 + 建议     │
└────────────────────────────────────────────────────────┘
```

### 交付层不要动的部分(已经做对了)
- 对比面板:产品卡纵向展示、原始参数独立分组、CSV带BOM头 — 符合你的偏好,保留。
- 建议横幅带具体参数值("最接近TPL8032, PSRR:70dB") — 符合你"交付够直接"铁律,保留。
- AND 搜索逻辑(防假阳性) — 保留。
- 确定性 Parser 优先 + LLM 兜底 — 思路对,保留,只是规则要归一到 schema。

## 落地优先级(综合三个决策)

```
P0 数据层根因修复(pymupdf坐标法)
   └─ 先做思瑞浦-模拟表头碎片(值已对,只修表头,风险低,30%→0%)
   └─ 同一套坐标逻辑推广到纳芯微(列转置)、裕太(记录型修列错位)

P1 单一真源归一(tag_schema.json) + route.ts 23补丁→规则解释器

P2 物料主数据层(Family/Material/Source三层) + 参数数值化

P3 交付层:近期结构化字段匹配;多厂商铺开后上 SQLite/DuckDB
```

每一步遵守铁律:零假阳性、系统性修复不打补丁、修完审计+压测拿数据证明、
参数=PDF原文不做语义推断、禁止对乱码产品自动打标签。
