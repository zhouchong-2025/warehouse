# Teampo / ChipSelect — 规模化与插件化架构

> 版本: 2026-06-08 | 用途: 资料库持续增长 + 对外作为"智能体可安装插件"的架构规划
> 配套: SYSTEM_ARCHITECTURE_HANDOFF.md(现状) / ARCHITECTURE_DECISIONS.md(物料/MinerU/交付)

---

## 第一部分:资料库增长的规模化架构

### 现状与增长外推(实测)

| 规模 | 全量 JSON 大小 | 前端全量 load 可行性 |
|------|---------------|---------------------|
| 2245 款(现在) | 1.2 MB | ✅ 可接受 |
| 5000 款 | ~2.6 MB | ⚠️ 开始变慢 |
| 10000 款 | ~5.3 MB | ❌ 首屏卡顿明显 |
| 30000 款 | ~15.8 MB | ❌ 不可用 |
| 100000 款 | ~52.8 MB | ❌ 浏览器直接崩 |

当前架构 `fetch 全量 JSON → 内存遍历 includes 匹配`,**拐点在 5000~10000 款**。
4 家厂商已 2245 款,再进 3~4 家国产模拟大厂(每家千款级)就会越过拐点。**必须提前规划。**

### 增长带来的三类压力

1. **数据量压力**:JSON 全量 load 不可持续(见上表)。
2. **数据异构压力**:每家厂商 PDF 布局不同(已证实 4 种布局),厂商越多适配越难。
3. **数据治理压力**:同族芯片跨厂商/跨册去重、参数口径对齐、版本更新(选型册每年更新)。

### 分层演进路线(按规模触发,不要一次到位)

```
阶段A 当前~5千款:  优化现有前端方案
  - 维持 JSON,但拆分: 按厂商/品类分片 JSON,按需加载(不再一次load全部)
  - 匹配从"字符串includes"升级为"结构化字段匹配"(参数数值化后)
  - 加客户端索引(如 lunr.js / minisearch 倒排索引)

阶段B 5千~5万款:  引入查询服务层
  - 数据落 SQLite(FTS5全文 + 数值列索引) 或 DuckDB(列存,擅长参数范围查询)
  - 前端不再全量load,改调 /api/search,服务端查询+分页
  - 这一步是"从静态文件到数据库"的关键跃迁

阶段C 5万款+:    专业检索栈
  - 结构化参数 → PostgreSQL(范围查询/事务/物料主数据)
  - 语义检索 → 向量库(pgvector / Qdrant),支持"找类似这颗料的替代品"
  - 二者混合检索(结构化过滤 + 语义召回)
```

### 数据治理:必须先于规模建立

规模化的真正难点不是存储,是**数据质量随规模衰减**。建议建立:

1. **物料主数据层(MDM)**:Family/Material/Source 三层(见 ARCHITECTURE_DECISIONS.md 决策一)。
   跨厂商、跨册的同族芯片统一,这是规模化的数据地基。
2. **入库流水线标准化**:新厂商 PDF → 归类到 4 种布局之一 → 坐标法提取 → 列级量纲校验 → 入 MDM。
   每一步可自动化、可审计(符合你的 validate.py→autofix.py 模式)。
3. **版本管理**:选型册每年更新(裕太 20250312、纳芯微 202510)。
   Source 层记录册版本,支持"这颗料在 2026 版参数变了"的对账。
4. **质量门禁**:入库前跑量纲校验 + 零假阳性审计,**不达标不进库**。防止脏数据稀释整体质量。

## 第二部分:插件化 —— 对外提供 API,成为智能体可安装的能力

### 目标

让 ChipSelect 从"一个网页"变成"一个其他 AI 智能体可以调用的选型能力"。
FAE 的智能体助手、客户的采购 bot、内部的报价系统,都能直接调用选型能力。

### 推荐:双协议对外(MCP + REST),共用一个查询内核

```
┌─────────────────────────────────────────────────────┐
│              查询内核 (Query Core)                     │
│   parseQuery(自然语言→结构化条件) + search(条件→产品)    │
│   ← 这是现有 query_parser + 匹配逻辑抽出来的纯函数        │
└───────────┬───────────────────────────┬─────────────┘
            │                           │
   ┌────────▼─────────┐       ┌─────────▼──────────┐
   │  MCP Server      │       │  REST API          │
   │  (给AI智能体)     │       │  (给传统系统/前端)   │
   │  stdio / HTTP    │       │  HTTP/JSON          │
   └──────────────────┘       └────────────────────┘
```

**关键设计原则:协议是薄壳,内核是同一套。** MCP 和 REST 都只是把内核函数暴露出去,
不重复实现选型逻辑。这样选型规则改一处,两个协议同时受益(避免又一次"双轨漂移")。

### 为什么选 MCP 作为智能体接入协议

- 你正在用 Hermes Agent,**原生支持 MCP 客户端**(配置 `mcp_servers` 即可自动发现工具)。
- MCP 是 Anthropic 推的智能体工具标准协议,Claude/Cursor/各类 Agent 都能直接装。
- "可安装的插件"在 AI 智能体语境下,标准答案就是 MCP server。

### MCP Server 应暴露的工具(建议)

| 工具名 | 输入 | 输出 | 用途 |
|--------|------|------|------|
| `search_parts` | 自然语言query / 结构化条件 | 匹配产品列表+评分 | 核心选型 |
| `get_part` | 型号 | 完整参数+来源 | 查规格书 |
| `compare_parts` | 多个型号 | 参数对比表 | 横向对比 |
| `find_alternatives` | 型号 | Pin-to-Pin/同族替代 | 国产替代 |
| `list_categories` | (无) | 品类树 | 探索可选范围 |

注: 在 Hermes 里这些工具会自动注册为 `mcp_chipselect_search_parts` 等。

### MCP 接入方式(两种)

**方式1: stdio(本地进程)** — 适合单机/内网部署
```yaml
# 使用方的 ~/.hermes/config.yaml
mcp_servers:
  chipselect:
    command: "node"
    args: ["/path/to/chipselect-mcp/server.js"]
```

**方式2: HTTP(远程共享)** — 适合多人/SaaS,推荐
```yaml
mcp_servers:
  chipselect:
    url: "https://chipselect.yourcompany.com/mcp"
    headers:
      Authorization: "Bearer <api_key>"
```

### REST API 应提供的端点(给非 AI 系统)

```
POST /api/v1/search          自然语言/结构化选型      → 产品列表
GET  /api/v1/parts/{pn}      单料详情                → 完整参数
POST /api/v1/compare         多料对比                → 对比表
GET  /api/v1/alternatives/{pn} 替代料                → 替代列表
GET  /api/v1/categories      品类树
GET  /api/v1/health          健康检查
```

### 对外 API 必须考虑的非功能项(规模化+插件化共同需要)

> ⚠️ 安全提醒:任何对外暴露的 API,认证和限流不是可选项。

1. **认证**:API Key / OAuth。不同调用方不同 key,可追溯、可吊销。
   **绝不要暴露无认证的公开端点**(尤其 search 会打到数据库)。
2. **限流(rate limit)**:按 key 限流,防止单个智能体把后端打挂。
3. **版本化**:`/api/v1/`。选型逻辑/数据结构会演进,版本化保证老调用方不被破坏。
4. **稳定的输出契约**:返回结构固定(产品schema),字段增量兼容,不随意改名。
   智能体依赖结构化输出,breaking change 会让所有接入方崩。
5. **数据来源标注**:每个返回的参数带 `source`(册名+页码),
   让调用方/客户能溯源——这是 B2B 选型场景的信任基础,也符合你的零假阳性铁律。
6. **错误语义清晰**:零结果时返回"为什么+替代建议"(你已有的建议逻辑),
   而不是冷冰冰的空数组——对智能体调用方尤其重要,它需要知道下一步怎么办。

### 落地顺序

```
1. 把选型逻辑抽成独立查询内核(纯函数,不依赖Next.js请求上下文)
   ← 这一步同时清理了现在 route.ts 把逻辑和HTTP耦合的问题
2. REST API 先行(现有 Next.js API route 稍改即可对外)
3. MCP Server 包一层(调用同一内核),先 stdio 本地验证,再上 HTTP
4. 加认证/限流/版本化
5. 在 Hermes 配置里自测: mcp_servers.chipselect → 让你的智能体直接选型
```

## 第三部分:规模化 × 插件化的架构协同

两件事不是独立的,共享同一套基础:

```
        ┌──────────────────────────────────────┐
        │   物料主数据层 MDM (Family/Material)   │  ← 规模化地基
        │   参数数值化 {value,unit,raw,source}   │
        └───────────────────┬──────────────────┘
                            │
        ┌───────────────────▼──────────────────┐
        │   查询内核 Query Core (纯函数)          │  ← 插件化地基
        │   tag_schema.json 单一真源驱动          │
        └───────┬───────────────────┬──────────┘
                │                   │
        ┌───────▼──────┐   ┌────────▼────────┐
        │ MCP / REST   │   │ 前端 Web         │
        │ (对外智能体)  │   │ (内部FAE)        │
        └──────────────┘   └─────────────────┘
```

**核心洞察:**
- 物料主数据层 + 参数数值化,**既是规模化的存储地基,也是 API 返回结构化数据的前提**。
- 查询内核抽成纯函数,**既解耦了前端,也让 MCP/REST 共用同一逻辑**。
- tag_schema.json 单一真源,**保证前端、MCP、REST 三个出口的选型规则永不漂移**。

所以这三件事(物料归一、查询内核抽离、单一真源)是**一鱼三吃**:
做好了同时解决"修修补补"、"规模化"、"插件化"三个问题。这才是值得投入的架构级改造。

## 优先级建议(融合所有讨论)

```
P0 数据层根因(pymupdf坐标法) — 先修思瑞浦-模拟表头碎片,证明30%→0%
P1 单一真源(tag_schema.json) + route.ts补丁→规则解释器
P2 物料主数据层(MDM) + 参数数值化  ← 规模化&插件化共同地基
P3 查询内核抽离(纯函数) + REST API 对外
P4 MCP Server 封装 + 认证/限流/版本化  ← 成为可安装插件
P5 规模到5千款触发: SQLite/DuckDB 查询服务层
```

铁律贯穿全程:零假阳性、系统性修复不打补丁、修完审计+压测、参数=PDF原文、
脏数据不进库、对外API必须认证限流。
