二、建议 Codex 优先处理的交付层问题（最小改动优先）

P0-1. 首页与 /products 页数据源不一致
现象:
- 首页 web/app/page.tsx 读取 /data/products_structured.json
- /products/page.tsx 读取 /data/products.json
- scripts/generate_data.py 生成的是 products.json

证据:
- web/app/page.tsx: fetch("/data/products_structured.json")
- web/app/products/page.tsx: fetch("/data/products.json")
- scripts/generate_data.py: output_path = OUTPUT / "products.json"

数量差异:
- products_structured.json: 1759
  - 3peak-analog 894
  - 3peak-auto 260
  - novosense 537
  - yutai 68
- products.json: 2483
  - 3peak-analog 825
  - 3peak-auto 380
  - novosense 1161
  - yutai 117

脏数据证据:
- products.json 中存在假型号/泛词条目: RS232, RS485

风险:
- 首页搜到的结果与 /products 页面结果口径不同
- /products 页面会展示低质量粗抽取结果
- 用户会感知为“首页有，列表页没有”或“列表页有脏型号”

建议修法:
A. /products 页面改为读取 products_structured.json，与首页统一单一真源
B. 如果 /products 需要轻量索引，单独从 structured 数据派生，不再使用 generate_data.py 的粗抽取 products.json
C. 若短期不删 products.json，至少将其明确标记为 legacy，不允许用户页继续消费

验收:
- 同一型号在首页与 /products?q=型号 的结果数一致
- /products 页面不再出现 RS232 / RS485 这类假型号


P0-2. 首页卡片 category badge 取法错误，参数会被当成品类
现象:
- 当前逻辑: 从 _features 去掉 grade 后，直接取最后一个 token 作为 categoryTag
- 导致大量产品卡片 badge 显示成 1通道 / Vin_36V / Vin_4.5V 一类参数标签

证据代码:
- web/app/page.tsx 约 506-514 行:
  const categoryTag = feat.split(" ").filter(...).pop() || "";

审计结果:
- suspicious_badge_count = 551

样例:
- LM2902A -> badge 可能为 1通道
- LM321A -> badge 可能为 Vin_36V
- TP1241 -> badge 可能为 Vin_4.5V

风险:
- 首屏信息误导
- 用户看不到真实品类，只看到参数碎片
- 后续“主推表高亮”若基于当前 badge 视觉体系，会进一步混乱

建议修法:
A. category badge 不要再用“最后一个 token”猜
B. 改成白名单品类优先匹配，例如只从 canonical category tags 中选：
   运放, 比较器, DCDC, LDO, SBC, CAN-FD, LIN, RS-485, RS-232, 数字隔离器,
   栅极驱动, 隔离栅极驱动, 非隔离栅极驱动, 马达驱动, 模拟开关, 电平转换, IO扩展,
   电流传感器, 温度传感器, 位置传感器, 电压基准, DAC, ADC, 隔离电源, 交换机, 网卡, 以太网供电 等
C. 若命中多个品类，按优先级取最具体的真实品类，而不是 capability tag
D. 若仍取不到，才 fallback 到 _section

验收:
- 随机抽 50 个卡片，badge 不应出现 Vin_/Vout_/Iout_/1通道/2口/100Mbps/SGMII 这类参数或媒介标签


P0-3. Suggestions 仅展示文本，不可点击采用
现象:
- suggestion 卡片只显示 s.text，没有交互
- 用户不能一键采用建议 query 重跑

证据代码:
- web/app/page.tsx 481-489 行（有结果时建议横幅）
- web/app/page.tsx 570-577 行（零结果建议卡）
- 都只是渲染文本，没有 onClick 使用 s.query

风险:
- 建议存在但不可操作
- 前端复测效率低
- 未来“主推表推荐型号高亮”无法自然接入建议流转

建议修法:
A. suggestion 卡增加“查看/采用此建议”按钮
B. 点击后执行:
   - setSearch(s.query)
   - setSearchTrigger(c => c + 1)
C. 若 suggestion.reason 是 too_many / no_match / best_alternative，可显示不同按钮文案
D. 若后续接主推表，可在 suggestion 采用后，把高亮逻辑一起触发

验收:
- 零结果与有结果两种 suggestion 卡都可以一键重搜
- 用户无需手工复制建议文案


P1-1. vendor 筛选逻辑未体现 3peak 聚合语义
现象:
- 当前首页筛选逻辑是纯 slug 精确匹配
- 没看到 3peak-analog + 3peak-auto 的聚合处理

证据代码:
- web/app/page.tsx 156-157 行:
  if (activeVendor && vendor !== activeVendor) continue;

风险:
- 用户选择某 vendor 时，3peak 两套 catalog 可能被拆开
- 后续做“主推表高亮”时，vendor 限定会有隐藏结果问题

建议修法:
A. 恢复/增加 vendorMap 聚合层
B. UI 可展示“3peak”总入口，内部同时匹配 3peak-analog + 3peak-auto
C. 若仍保留两个子入口，也要保证语义查询或推荐逻辑可选择“聚合 3peak”模式

验收:
- 以“3peak 隔离驱动/3peak CAN”类查询测试，不会因为 vendor filter 把另一半 catalog 藏掉


P1-2. /products 页面本身过于简陋，不适合作为正式交付入口
现象:
- /products 只按 part_number contains 过滤
- 不展示结构化参数
- 数据源还是 legacy products.json

风险:
- 用户从首页跳过去后，感知质量下降
- 像一个调试页，不像正式页面

建议修法:
A. 若保留 /products，就切 structured 数据并至少展示 vendor + 核心参数
B. 若无明确用途，建议降级为内部调试页，不暴露给用户主流程
